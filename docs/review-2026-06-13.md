## 0. TL;DR — the high-value items

If we only did a handful of things, these give the most return for the least
code:

1. **Advertise EWMH support** — `_NET_SUPPORTING_WM_CHECK` + `_NET_SUPPORTED`
   (§B1). Cheap, and the single most likely lever on the GTK file-picker delay
   (§G2).
2. **Handle `ConfigureRequest`** (§A1). Today it's silently dropped; this breaks
   client-initiated sizing (dialogs, pickers) and is a likely co-cause of "weird
   picker behaviour."
3. **Fix the duplicate `on_unmap_notify`** (§A2) — windows never get
   `mapped=false`, so closed/hidden windows linger in the tiling.
4. **Fix `toggle_maximize` restore typo** (§A3) — unmaximize currently doesn't
   restore.
5. **Strip the `pp`/`p` debug spam** now that `$logger` exists (§C1). Biggest
   single readability + log-noise win.
6. **Fix the multi-monitor association internals** (§D) — the current two-way
   monitor↔desktop sync is the most fragile new code. Moving desktops freely
   between monitors is a hard requirement, so the fix is to keep that UX but make
   the association single-owner (§D3), not to adopt bspwm's ownership model.

## A. Correctness bugs (current WM)

### A1. No `ConfigureRequest` handler — client resizes are silently dropped *(high)*
`root.select_input` includes `SubstructureRedirectMask` (`wm.rb:29`), which means
the X server **redirects** `ConfigureRequest` to the WM and does **not** apply
it. There is no `on_configure_request` (`grep` confirms none). So when a client
asks to resize/move/raise itself, nothing happens — the request is consumed and
dropped.

Consequences:
- Floating dialogs / file pickers that map small then request their real size
  never grow.
- A window that configures itself *before* mapping can be seen by `adopt` at its
  pre-configure size; `adopt` then drops anything with `width < 2 || height < 2`
  (`wm.rb:299`) — i.e. the window is never managed at all.

Minimum correct behaviour (ICCCM §4.1.5): for windows we don't tile, honour the
request and synthesize a `ConfigureNotify`; for tiled windows, ignore the
geometry but still send the synthetic `ConfigureNotify` so the client isn't left
waiting. ~15 lines.

```ruby
def on_configure_request(ev)
  w = @windows[ev.window]
  if w.nil? || w.floating?
    # honour position/size for unmanaged + floating
    configure_from_request(ev)      # x/y/width/height/border/stack from value_mask
  else
    w.send_synthetic_configure      # ICCCM: tell the client its real geometry
  end
end
```

### A2. Duplicate `on_unmap_notify` — second definition wins *(high)*
`wm.rb:475` `def on_unmap_notify(ev) = (window(ev.window)&.mapped = false)` is
**overwritten** by `wm.rb:550` `def on_unmap_notify(ev) = window(ev.window)&.desktop&.update_layout`.
Ruby keeps the last definition, so `mapped` is *never* set back to false on
unmap. Windows that unmap without destroying (minimise, hide-to-tray, some
dialog close paths) stay `mapped=true` and keep a tile slot until a
`DestroyNotify` arrives.

Fix: merge into one handler that does both.
```ruby
def on_unmap_notify(ev)
  w = window(ev.window) or return
  w.mapped = false                  # the setter already calls desktop.update_layout
end
```
(Beware the WM's own hide/show uses move-offscreen, not unmap — good, so this
won't fight your hiding scheme. Worth confirming in the harness.)

### A3. `toggle_maximize` restore typo — unmaximize doesn't restore *(high)*
`window.rb:200`: `@real_geom = @old_geom if @old_geom` — note `@real_geom`
(underscore) vs the real ivar `@realgeom`. The next line restores
`resize_to_geom(@realgeom)`, but `@realgeom` was set to the monitor geometry by
`maximize` (via `resize_to_geom`, `window.rb:173`). Net effect: unmaximize leaves
the window maximized; `@real_geom` is written once and never read.

Fix: `@realgeom = @old_geom if @old_geom` (drop the stray underscore), or restore
directly from `@old_geom`.

### A4. `set_current_desktop` writes the wrong monitor's property *(medium)*
`set_current_desktop` (`wm.rb:63`) builds
`"_NET_CURRENT_DESKTOP_MONITOR_#{active_monitor.id}"`, but it's called from
`change_desktop_on_monitor(curr_monitor, …)` (`wm.rb:436`) for a *specific*
monitor that may not be the one under the pointer. If you switch a monitor's
desktop while the pointer is elsewhere, the per-monitor property for the *wrong*
monitor is updated — and Rebar reads exactly this property (§F). There's already
a `# FIXME: pass monitor id` at `wm.rb:435`. Pass the monitor through.

### A5. Dialog/utility windows are re-created on every event *(medium)*
`adopt` early-returns for popup/menu/dock/tooltip/**dialog**/splash/utility
(`wm.rb:277–289`) **without** storing them in `@windows`. So `window(wid)` →
`adopt` makes a *brand-new* `Window` object on every subsequent reference,
discarding per-window state (`@realgeom`, `@maximized`, `@hidden`). For a
`_NET_WM_WINDOW_TYPE_DIALOG` (which many file pickers are) this means the WM
never has a stable handle on it. Likely interacts badly with §A1.

Options:
- (a) Store them too (`@windows[wid] = w`) but mark `floating` and skip tiling.
  Simplest; gives stable identity + client-list membership.
- (b) Keep ignoring truly transient override-redirect surfaces (menus/tooltips
  are usually override-redirect and we never see them anyway), but always manage
  DIALOG/UTILITY/SPLASH. This is closest to what dwm/bspwm do (they float these,
  but still manage them).

Recommend (b).

### A6. `Window#eql?`/`==` without `#hash` *(medium, latent)*
`window.rb:10–11` override `eql?`/`==` by `wid` but not `hash`. `TiledLayout#call`
does `windows - @root.children` (`tiled.rb:54`) — `Array#-` uses `hash` **and**
`eql?`. Two `Window` instances with the same `wid` hash differently, so set-diff
treats them as distinct. Today identity is usually stable (one object per wid in
`@windows`), which masks it — but §A5 already creates duplicate objects, so this
is live, not theoretical. Add:
```ruby
def hash = wid.hash
```

### A7. `--debug` flag never triggers; `--drb` blocks the event loop *(medium)*
`rubywm.rb:60` `if ARGV.shift == "--drb"` then `rubywm.rb:67` `if ARGV.shift == "--debug"`.
Two consecutive `ARGV.shift`s: with a single `--debug` arg, the first shift
consumes it (compares to `--drb`, false), the second shift returns `nil` — so
`--debug` alone is dead. And the `--drb` branch ends with `gets` (`rubywm.rb:64`)
which blocks *before* the main event loop, so `--drb` mode appears to never pump
X events (unless stdin is already closed). Replace with a tiny real arg parse
(set booleans, then run loop once). Trivial but currently misleading.

### A8. NumLock in the button grabs *(medium)*
`wm.rb:23–24` grab `Mod2|Mod4`. `Mod2` is conventionally **NumLock** — so those
two grabs only fire when NumLock is *on*. Lines 25–26 grab plain `Mod4` (Super),
which is what you actually want. The clean idiom (dwm does this) is to grab your
modifier against *every* combination of the lock modifiers and otherwise ignore
them:
```ruby
LOCKS = [0, X11::Form::Mod2, X11::Form::Lock, X11::Form::Mod2|X11::Form::Lock]
[1,3].each { |btn| LOCKS.each { |l| root.grab_button(true, mask, :async, :async, 0, 0, btn, X11::Form::Mod4|l) } }
```

### A9. `change_desktop_on_monitor` two-way swap is fragile *(medium)*
The desktop-swap path (`wm.rb:412–419`) sets `current_monitor_desktop.monitor = nil`
and then *recurses* into `change_desktop_on_monitor(other_monitor, …)`, while
`Desktop#monitor=` (`desktop.rb:9`) and `Monitor#active_desktop=` (`monitor.rb:7`)
call **each other** with guard clauses. This mutual write-through is hard to
reason about and is the most likely place for a subtle "desktop shows on two
monitors / on none" state bug. See §D for the structural fix (single owner of
the association).

### A10. Smaller ones
- `window.rb:129` `net_wm_desktop=d.id` parses as assignment to a throwaway
  **local variable** (no receiver) — dead code; the real work is the
  `change_property` on the next line. Delete.
- `Monitor#inspect` (`monitor.rb:36`) and `Desktop#inspect` (`desktop.rb:32`)
  reference `@name`/`.name` on `Monitor`, which has no `name` — always `nil` in
  logs. Either add `name:` or drop it from `inspect`.
- `process_monitors(config)` (`wm.rb:86`) ignores its `config` argument.
- Logger severity colours look swapped: `ERROR`→green(32), `INFO`→yellow(33)
  (`rubywm.rb:25–31`). Mildly alarming-in-reverse.

## B. EWMH / ICCCM adherence gaps

The WM currently sets: `_NET_NUMBER_OF_DESKTOPS`, `_NET_CURRENT_DESKTOP`,
`_NET_ACTIVE_WINDOW`, `_NET_CLIENT_LIST`, and per-window `_NET_WM_DESKTOP`. Good
start. The notable omissions:

### B1. `_NET_SUPPORTING_WM_CHECK` + `_NET_SUPPORTED` *(high)*
Neither is set (`grep` confirms). Per EWMH these are how a client decides "is an
EWMH-compliant WM running, and what does it support." Consequences when missing:
- `wmctrl`, pagers, and some toolkits conclude there's *no* EWMH WM.
- GTK/Qt take conservative/fallback paths for fullscreen, maximize, and
  (relevantly) some startup synchronisation — see §G2.

Implementation is ~10 lines: create a small never-mapped child window, set
`_NET_SUPPORTING_WM_CHECK` → that window on **both** root and the child, set
`_NET_WM_NAME` ("rubywm") on the child, and publish `_NET_SUPPORTED` listing the
atoms we actually honour. Keep `_NET_SUPPORTED` honest — only list what we
implement.

### B2. `_NET_WORKAREA` / `_NET_DESKTOP_GEOMETRY` / `_NET_DESKTOP_VIEWPORT` *(medium)*
Apps that maximize themselves read `_NET_WORKAREA` to know the usable rectangle
(screen minus struts). We don't publish it, so self-maximizing apps guess. This
also connects to the hardcoded `@geom.height -= 30` bar fudge (`tiled.rb:24`) and
the Rebar `x=1` workaround — see §B5 and §F.

### B3. ICCCM `WM_STATE` *(medium)*
We never set `WM_STATE` (Withdrawn/Normal/Iconic) on managed windows. Pagers,
taskbars, and some apps test it to decide whether a window is "really" managed.
Cheap to set on map/unmap.

### B4. Closing windows ignores `WM_DELETE_WINDOW` *(medium)*
`on_wm_delete_window` (`wm.rb:592`) just `@focus.destroy` — it force-kills the X
resource instead of sending the `WM_DELETE_WINDOW` ClientMessage to clients that
advertise it in `WM_PROTOCOLS` (with `XKillClient` fallback). As-is, "close"
loses unsaved work in well-behaved apps. The FIXME at `wm.rb:587–589` already
notes the intended behaviour; it's worth doing. ~15 lines, including reading
`WM_PROTOCOLS`.

### B5. Struts not honoured (`_NET_WM_STRUT_PARTIAL`) *(medium)*
Docks (Rebar) should reserve screen edge space; the WM should subtract that from
the tiling area and from `_NET_WORKAREA`. Today the bar height is hardcoded
(`tiled.rb:22–24`, "we shouldn't just assume the height here"). Reading
`_NET_WM_STRUT_PARTIAL` from dock windows and computing per-monitor work areas
removes the magic 30 *and* the Rebar offset hacks in one move. (Rebar must also
*set* the strut — it currently doesn't; §F.)

### B6. `_NET_CLIENT_LIST_STACKING`, `_NET_DESKTOP_NAMES` *(low/medium)*
- `_NET_CLIENT_LIST_STACKING` (bottom→top): pagers/alt-tab tools want it.
- `_NET_DESKTOP_NAMES`: **Rebar reads it** but the WM doesn't set it — Rebar
  currently writes the names itself (`setnames.rb`). The WM owns desktop
  identity; it should publish the names (from `config.yml`, which already has a
  `name:` field). Then Rebar just reads. Removes a moving part from Rebar.

### B7. `_NET_WM_STATE` handling is partial *(low)*
`on_net_wm_state` (`wm.rb:571`) only maps `FULLSCREEN` → `toggle_maximize` and
ignores the `action` field (add/remove/toggle, `_NET_WM_STATE` values 0/1/2) and
the MAXIMIZED_VERT/HORZ atoms. It also conflates *fullscreen* (cover the whole
monitor, above docks, no border) with *maximize* (fill the work area, keep
struts). Worth separating eventually; low priority.

## C. Cleanliness & compactness

You want to *stay* ~1.5 KLOC while getting cleaner. Most of the bulk in the new
code is debug scaffolding and defensive duplication, so this is mostly deletion.

### C1. `pp`/`p` debug spam *(high — pure deletion)*
Now that `$logger` exists, the bare `pp`/`p` calls are both noise and a small
perf tax on hot paths. Concentrations:
- `floating.rb:35,52,61,66,72,80` — ~6 `pp` per window placement.
- `window.rb:68,77,91,104,112` — `p` on every `show` of a floating window.
- `wm.rb:87,89,113` (`process_monitors`), `wm.rb:129,139,142,145,151,172`
  (`active_monitor`/`monitor_for_window`), `wm.rb:573` (`on_net_wm_state`).

Recommendation: delete outright, or convert the genuinely useful ones to
`$logger.debug { … }` (block form so they cost nothing at INFO level). This alone
shrinks `floating.rb` by ~30%.

### C2. `FloatingLayout#place` is doing too much *(medium)*
`floating.rb:28–82` is ~55 lines, half of it `pp` and commented-out
alternatives, plus a self-admitted "geom is the window dimensions, not monitor
dimensions — bug somewhere" (`floating.rb:47`). It also force-reassigns
`w.desktop = @wm.active_monitor.active_desktop` (`floating.rb:75`), overriding the
caller's intent, and `current_geometry` ignores `@desktop` and always uses the
active monitor (`floating.rb:26`) — so placing onto a *non-active* monitor's
desktop uses the wrong rectangle. A compact rewrite that (a) trusts
`@desktop.geometry`, (b) centres only when the client gave x/y == 0, (c) drops
the logging, should be ~15 lines and fix the "geom is wrong" confusion.

### C3. Window-type classification is duplicated and stringly *(medium)*
The "is this a floaty/special type" decision exists in three places with three
spellings: the big `||` chain in `adopt` (`wm.rb:277–285`), `Window#stack`
(`window.rb:156–165`), and `special?`/`dock?`/`desktop?` (`window.rb:147–149`).
Consolidate into one place on `Window`, backed by a cached `Set` of atom ids:
```ruby
FLOAT_TYPES = %i[_NET_WM_WINDOW_TYPE_DIALOG _NET_WM_WINDOW_TYPE_UTILITY
                 _NET_WM_WINDOW_TYPE_SPLASH _NET_WM_WINDOW_TYPE_TOOLTIP
                 _NET_WM_WINDOW_TYPE_MENU  _NET_WM_WINDOW_TYPE_POPUP_MENU
                 _NET_WM_WINDOW_TYPE_NOTIFICATION]
def floaty_type? = FLOAT_TYPES.any? { type == dpy.atom(_1) }
```
Fewer lines, single source of truth, and it makes §A5 a one-liner. (Atom ids are
constant per session — intern once into a hash to avoid repeated `dpy.atom`
round-trips on hot paths.)

### C4. Magic numbers → named/derived *(low)*
`GAP=64` (`tiled.rb:2`), bar `30` (`tiled.rb:24`), node single-child `0.3`/`0.15`
(`node.rb:73–74`), floating `0.7` (`floating.rb:49–50`), move step `20`
(`wm.rb:668–671`). Fine as constants, but the bar `30` should come from struts
(§B5) and the `0.3/0.15` "reserve space around a lone window" is really a layout
policy worth a name.

### C5. The `layout` shim logs three warnings + a backtrace *per call* *(medium)*
`wm.rb:223–228` warns (×3 + `caller[0..9]`) every time `layout` is called — and
it's still called by `layout_for` (`wm.rb:237`), `on_rwm_shift_direction`
(`wm.rb:634`), and `on_rwm_swap_nodes` (`wm.rb:647`). So normal operation spews
warnings. Two of those callers are genuinely multi-monitor-unsafe (they should
use `@focus.desktop.layout`); `layout_for` calls it for the non-floating branch
and then immediately re-fetches geometry. Either finish the migration (make the
three callers use the focused window's desktop layout) and delete the shim, or
drop the per-call logging and keep it as a documented fallback. Right now it's
the worst of both.

## D. Multi-monitor model — the structural decision

**Hard constraint (confirmed):** desktops must be movable between monitors
freely — you do this regularly and it's essential. So the bspwm/i3 "a desktop is
owned by one monitor" model is **off the table**; it's listed below only to mark
it explicitly rejected. The work here is to keep the float-across-monitors UX and
fix the *internals* that make it fragile (§A4, §A9).

### Option D1 — "Desktops float across monitors" (today's model + behaviour)
All N desktops exist globally; any monitor can display any desktop; switching to
a desktop already shown elsewhere swaps the two monitors' desktops. State lives
in a **two-way** association (`Monitor#active_desktop` ↔ `Desktop#monitor`), kept
in sync by mutual setters with guards.

- 👍 Any desktop reachable from any monitor — the required UX.
- 👎 The two-way sync is fragile (the recursion in §A9). "Current desktop" is
  ambiguous (`current_desktop_id` vs per-monitor). Needs the non-standard
  `_NET_CURRENT_DESKTOP_MONITOR_<n>` atom — but that's inherent to per-monitor
  desktops and fine to keep as a documented RubyWM extension.

### Option D3 — keep D1's UX, make the association single-owner *(recommended)*
Same behaviour as today, but stop storing the association on *both* sides.
`Monitor` owns `active_desktop`; `Desktop#monitor` becomes a *derived read-only*
lookup (`@wm.monitors.find { _1.active_desktop == self }`) instead of a stored,
mutually-written field with guard clauses. The swap then becomes a single
explicit operation ("set monitor A's active desktop to D; if D was showing on
monitor B, set B's active desktop to A's old desktop") with no mutual callbacks.
Removes the §A9 recursion and the §A4 ambiguity without changing what you can do.

- 👍 Keeps free desktop movement; collapses the fragile two-way sync to one
  direction; "which monitor shows desktop D" has exactly one answer.
- 👎 A small rewrite of `Desktop#monitor=` / `Monitor#active_desktop=` and the
  swap path in `change_desktop_on_monitor`. Behaviour-preserving, so testable by
  asserting the same property/visibility outcomes before and after.

### Option D2 — "Desktops belong to a monitor" (bspwm/i3) — *rejected*
Listed for completeness only. Each desktop owned by one monitor; switching cycles
within a monitor's own desktops. **Not acceptable** — it forbids moving a desktop
to another monitor, which is an essential workflow. Do not pursue.

Recommendation: **D3.** Keep the behaviour, kill the two-way setter, document
`_NET_CURRENT_DESKTOP_MONITOR_<n>` as an intentional RubyWM extension (§E1).

A separate observation: `active_monitor` (`wm.rb:135`) does a synchronous
`query_pointer` round-trip on *every* call, and it's called from `change_desktop`,
`map_window`, `adopt`, `set_current_desktop`, and `FloatingLayout#current_geometry`.
bspwm tracks the focused monitor from enter/focus events instead of polling the
pointer. Consider caching "focused monitor" and updating it on `EnterNotify`/
`FocusIn`, falling back to a pointer query only when stale.

## E. IPC strategy (X11-first, D-Bus optional)

Today: **commands** arrive as ClientMessages (`_NET_*` and custom `_RWM_*`);
**state** is partly published as root properties; a DRb Unix socket exists for
debug only (`drb.rb`). Your stated preference — X11 over a custom socket, D-Bus
only if compact — maps cleanly onto the EWMH idiom. Options, roughly in order of
how much I'd lean on each:

### E1 — Double down on "properties out, ClientMessages in" *(recommended baseline)*
This is the EWMH contract and what Rebar already does: the WM publishes all state
as root/window properties and bumps them on change; clients `select_input`
`PropertyChangeMask` on root and react to `PropertyNotify` (no polling); clients
send commands as ClientMessages. Action items: finish the published-state set
(§B), and tidy the `_RWM_*` command surface (below). No socket, no D-Bus.

- 👍 Zero new dependencies; pagers/tools already speak it; survives WM restart
  (state is in the server).
- 👎 ClientMessage is fire-and-forget, max 5×32-bit words, **no return value** —
  fine for commands, useless for queries.

Tidy-ups within E1:
- `_RWM_MOVE_TO_MONITOR`/`_RWM_FOCUS`/`_RWM_MOVE` encode the direction as an
  **atom name** the WM then `get_atom_name`s back (`wm.rb:606,624,662`). The
  `on_rwm_move` FIXME (`wm.rb:658`) already suggests passing integer offsets in
  `data[]` instead — do that across the `_RWM_*` set; it's faster and removes a
  round-trip.
- Document the `_RWM_*` protocol in the README (atom, data layout, semantics)
  the same way EWMH documents its messages.

### E2 — Property-based request/response for *queries* (pure X11)
When you eventually want `bspc query`-style introspection (list windows, dump the
layout tree), a ClientMessage can't return data. The pure-X11 answer: client
writes a request id to a property on its own window and sets a "request" atom; WM
writes the reply onto a property the client watches. Works, but it's clunky and
racy to get right.

- 👍 No new deps; still "just X11."
- 👎 Awkward, easy to get the round-trip/ownership wrong; really only worth it if
  queries stay rare.

### E3 — A *compact* D-Bus service for the rich/query API
Expose a single `org.rubywm` service with a handful of methods
(`SwitchDesktop`, `FocusDir`, `Query`) and signals (`DesktopChanged`). Keep
commands *also* available as ClientMessages (E1) so the EWMH world keeps working;
D-Bus is purely the "I want a return value / structured query" channel.

- 👍 Real return values, introspection, signals; integrates with the session bus
  you already export in `.xinitrc`.
- 👎 D-Bus ceremony (you said you dislike it). Mitigations: a tiny hand-rolled
  client over the session bus is ~100–150 lines; or depend on a small gem. Either
  way it's more than E1/E2.

### E4 — Keep DRb, but for power-users not just debug
`drb.rb` already gives you a full Ruby object over a Unix socket. You explicitly
prefer X11 over a custom socket, so I'd **not** promote DRb to the primary API —
but it's a great scripting/debug back door to keep as-is.

Recommendation: **E1 as the contract**, add **E2** only if/when a real query need
appears, and treat **E3** as the escape hatch if you ever want structured replies
without the property dance. Avoid making DRb load-bearing.

## F. Rebar integration (`~/Desktop/Projects/Rebar`)

Rebar is mostly doing the right EWMH things. Per your instruction I separate
*Rebar's own bugs* from *things the WM should provide*. **Correction to the
explorer's report:** sending `_NET_CURRENT_DESKTOP` with
`SubstructureNotifyMask | SubstructureRedirectMask` (`lib/ewmh.rb:40–42,52–54`)
is **correct** — EWMH specifies exactly that mask for root-directed messages.
Not a bug; don't change it.

What Rebar depends on (all reasonable):
- Reads `_NET_NUMBER_OF_DESKTOPS`, `_NET_CURRENT_DESKTOP`, `_NET_DESKTOP_NAMES`,
  and the custom `_NET_CURRENT_DESKTOP_MONITOR_<n>` (`lib/ewmh.rb:24–36`).
- Event-driven via root `PropertyChangeMask` (`rebar.rb:65`) — good, no polling
  (the polling loop in `desktop_switcher.rb:71–82` is dead code; see below).
- Sets `_NET_WM_WINDOW_TYPE_DOCK` (`lib/ewmh.rb:17–20`) and uses
  `override_redirect` (`rebar.rb:52`).

Bugs/cleanups **in Rebar**:
- **F1.** Dead polling loop in `DesktopSwitcher.run` (`desktop_switcher.rb:71–82`)
  — defined, never used, confusing. Delete.
- **F2.** `x = 1 + screen.x_org` with comment "my wm refuses to honour x=0"
  (`rebar.rb:40`). Since Rebar is `override_redirect`, the WM shouldn't be
  touching it at all (`adopt` bails on override-redirect, `wm.rb:296`). Once the
  WM advertises support and handles ConfigureRequest (§A1/§B1), re-test x=0 and
  drop the `+1`. This is a Rebar-side workaround for (probably) an old WM bug.
- **F3.** No `_NET_WM_STRUT_PARTIAL` (`rebar.rb:37` "FIXME: Set *strut*"). For the
  WM to reserve bar space cleanly (§B5), Rebar must publish its strut. This is
  the Rebar half of removing the hardcoded `30` in the WM.
- **F4.** The null-byte-padding retry loop when writing properties
  (`changename.rb:22–37`, `setnames.rb:12–14`) is a cargo-culted workaround for a
  property-encoding quirk in the X11 binding. Worth root-causing in `pure-x11`
  rather than padding until it sticks — it'll bite again.

WM-side items that *help* Rebar without adapting to its bugs:
- Publish `_NET_DESKTOP_NAMES` from the WM (§B6) so Rebar stops owning names.
- Fix the per-monitor property to track the right monitor (§A4) — Rebar's

## G. Session integration & `.xinitrc` (incl. the file-picker delay)

Your `~/.xinitrc` starts: sxhkd, rubywm, two Rebars, Dwin (`bin/desktop`),
`bin/todo`, randombg, compton, nm-applet, pnmixer.

### G1. Blocking launches stall the rest of the session *(high, easy)*
- `/home/vidarh/bin/todo` (the line with your own "will this open reliably?"
  comment) is **not** backgrounded. If `todo` is a GUI that stays open, *every
  line after it* — randombg, compton, nm-applet, pnmixer — never runs until todo
  exits. That alone could explain "things feel half-started."
- `randombg 2>&1 | logger -t randombg` is also not backgrounded. If it ever
  blocks, it stalls compton + applets behind it.
- Fix: background anything that isn't instantaneous (`( … ) &`), and only the
  final long-lived "anchor" should hold the session open (you already have the
  `while true; wait; sleep 5` keep-alive at the end).

### G2. The GTK file-picker delay — likely causes, ordered *(high)*
This is almost always *session plumbing*, not the WM proper. Your read — that
it's a failing D-Bus/portal probe rather than the WM — matches the usual
signature of this delay (a long, *fixed* stall on first picker open, then fine
afterwards = a name request timing out on the bus). Ordered accordingly:

1. **xdg-desktop-portal probe timeout *(most likely)*.** GTK 3.24+/GTK4 file
   choosers try the portal (`org.freedesktop.portal.Desktop`) over the session
   bus *first*. With no portal running, the D-Bus daemon tries to **activate**
   one; if a `.service` file exists but the backend can't start (no portal
   backend installed, or it needs an env/`XDG_CURRENT_DESKTOP` it doesn't have),
   you get the classic ~25 s activation timeout before GTK falls back to its
   in-process chooser. Two fixes, pick one:
   - (a) Force the in-process chooser, skip the bus entirely: export
     `GTK_USE_PORTAL=0` (GTK3) and for GTK4 ensure no portal is advertised
     (`unset GDK_DEBUG`; GTK4 only uses the portal if it answers). Best for a
     minimal WM.
   - (b) Actually run a portal: launch `/usr/lib/xdg-desktop-portal` with a
     backend (`xdg-desktop-portal-gtk`) and set `XDG_CURRENT_DESKTOP` so it knows
     which backend to pick. More moving parts; only worth it if you want portal
     features (screenshots, sandboxed file access).
2. **Activation environment not propagated.** You export
   `DBUS_SESSION_BUS_ADDRESS`/`XDG_RUNTIME_DIR` in `.xinitrc`, but nothing runs
   `dbus-update-activation-environment --systemd DISPLAY XAUTHORITY
   XDG_CURRENT_DESKTOP`. Without it, *any* D-Bus-activated service (the portal
   above, gvfs) starts with a broken environment and is more likely to hang or
   mis-probe. Adding this one line early often fixes the portal stall at the
   source rather than papering over it. (And set `XDG_CURRENT_DESKTOP` to
   something — even a made-up value — so portal/backends stop probing for it.)
3. **gvfs activation.** The picker's "Recent" / "Other Locations" can trigger
   `gvfsd`/`gvfs-*` D-Bus activation; if those can't start cleanly you get a
   secondary stall. Same root cause as #1/#2 (bus activation in a thin session).
4. **AT-SPI accessibility bus wait.** GTK apps also try `org.a11y.Bus`; with no
   a11y stack this can add a smaller delay. `NO_AT_BRIDGE=1` (GTK3) /
   `GTK_A11Y=none` (GTK4) removes it. Cheap to add alongside the above; usually a
   smaller contributor than the portal probe.
5. **WM not advertising EWMH (§B1).** Low probability for the *picker* delay
   specifically, but worth doing anyway.

Diagnostic to confirm it's the bus probe: run the picker app under
`strace -f -e trace=connect,poll,recvmsg` (or `DBUS_VERBOSE`/`G_MESSAGES_DEBUG`)
and watch for a long `poll` on the session-bus socket — a ~25 s wait there is the
portal/gvfs activation timeout, not the WM. `busctl --user list` while it hangs
shows whether a `org.freedesktop.portal.*` activation is stuck.
Fastest thing to try first: export `GTK_USE_PORTAL=0` and add
`dbus-update-activation-environment …` to `.xinitrc` — two lines, and it
addresses exactly the failing-probe case you suspect.

### G3. GNOME / display-manager coexistence *(medium, optional)*
"Minimal GNOME integration" — the lightest useful steps:
- Ship an Xsession `.desktop` so a display manager can offer "RubyWM" as a
  session (a 5-line `/usr/share/xsessions/rubywm.desktop` pointing at a wrapper

- If you want GNOME *components* (settings daemon, keyring) without GNOME's WM,

## H. Dwin / desktop window (`~/Desktop/Projects/Dwin`)

**Two clarifications that flip the earlier framing of this section:**

1. **Dwin must live on exactly one desktop — not all of them.** Per the design
   intent, the Dwin desktop window appears *only* on its configured desktop
   (desktop 10 here, the floating one). Confining a `_NET_WM_WINDOW_TYPE_DESKTOP`
   window to a single desktop is a **deliberate feature** and one of the
   motivations for leaving bspwm, which insists on making desktop windows
   sticky/global. So the goal is the opposite of "show it everywhere": honour
   `_NET_WM_DESKTOP` for desktop-type windows like any other window, just keep
   them lowered and unfocusable.
2. **Dwin does not draw the wallpaper.** `randombg` sets the root-window
   background (root pixmap), which is why the *image* shows on every desktop —
   that's independent of Dwin and working as intended. Dwin only draws the
   desktop *icons / interaction surface*, which belongs on desktop 10 only.

So this is a **regression, not a model problem**: the single-desktop desktop
window "used to work" and the multi-monitor refactor broke it. `_NET_WM_DESKTOP =
[9]` (0-indexed → desktop 10) encodes the *correct intent*; the value isn't the
bug.

Where the confinement most likely broke (to investigate against the harness — a
DESKTOP-type client on `:N`, then switch desktops and screenshot):
- **Desktop-type windows are forced `floating = true`** in `adopt`
  (`wm.rb:290–292`) and routed through `FloatingLayout`. `FloatingLayout#place`
  then **reassigns** the window's desktop to the active monitor's active desktop
  (`floating.rb:75` `w.desktop = @wm.active_monitor.active_desktop`) — which would
  yank Dwin off desktop 10 onto whatever's focused. Strong suspect.
- `Window#show` runs floating windows through monitor-relocation math
  (`window.rb:62–107`). For the desktop window this relocation is *wanted* (it
  must follow desktop 10 to whichever monitor shows it) — but the semantics are
  wrong: it should **fill the hosting monitor**, not preserve a floating
  size/relative position. So the issue isn't that it relocates, it's that it
  relocates *as a floating window* instead of *as a full-monitor backdrop*.
- The `desktop: true` flag on desktop 10 in `config.yml` is **read nowhere** —
  `process_config` never consumes it (`wm.rb:201` only checks `layout`). It looks
  like the intended "this desktop hosts the desktop window" marker that was never
  wired up (or lost). See H1.

Recommended direction:
- **H1 (WM-side):** a `_NET_WM_WINDOW_TYPE_DESKTOP` window stays **pinned to its
  configured desktop** (the one named by `_NET_WM_DESKTOP`) and is **never
  reassigned to another desktop** — that reassignment (`floating.rb:75`) is the
  regression. It is shown/hidden with that desktop and, when shown, **follows the
  desktop to whichever monitor currently displays it and fills that monitor's
  geometry** (so switching to desktop 10 on monitor B makes the desktop window
  appear, full-size, on B). Its *only* other special-casing is: **always
  lowered** (below every other window) and **never focused**. (Forward-looking:
  this lowered backdrop is also where you want **iconified windows** to live, so
  keep the design open to it hosting icons rather than treating it as inert.)
  This single-desktop-but-monitor-following behaviour is exactly what bspwm
  wouldn't let you do.
- **H2 (config wiring):** the desktop assignment shouldn't be hardcoded in Dwin
  (`[9]`) — make it configurable. The `desktop: true` flag already on desktop 10
  in `config.yml` is the natural place: the WM consumes it, treats that desktop as
  the host for desktop-type windows, and **publishes which desktop hosts the
  desktop window** (a `_RWM_*` root property) so Dwin reads it instead of
  hardcoding. Removes the brittle hardcode and survives renumbering.

  - **All-desktops option (opt-in, with a caveat).** It can make sense to let a
    desktop window be configured as visible on *all* desktops (EWMH
    `_NET_WM_DESKTOP = 0xFFFFFFFF`) — but only for **floating** desktops. On a
    **tiling** desktop a screen-covering desktop window has no sensible place in
    the layout: it either fights the tiler or has to be special-cased as a
    non-tiled layer beneath the tiles, which adds complexity for little benefit.
    So if we expose this, scope it to floating desktops and either ignore or warn
    when a tiling desktop is configured with an all-desktops desktop window.
    Default stays single-desktop (the current, working intent).

  This is config/policy design, not a blocker — settle the schema (`desktop:
  true` vs an explicit `desktop_window: <id|all>` key) when we tackle H1.

Bugs **in Dwin** (don't adapt the WM to these):
- **H3.** Selects `KeyPressMask` on the **root** window (`desktop.rb:94`) to grab
  the Menu/Option key. That collides with the WM (which also selects on root) and
  with sxhkd. A desktop window shouldn't be reading root key events; it should
  grab on its own window or leave global hotkeys to sxhkd. Likely source of
  flaky key behaviour.
- **H4.** Single-monitor assumptions: "blindly assuming all monitors are the same
  size" (`desktop.rb:61`), `x % 1920` hardcode for click coordinates
  (`desktop.rb:260`, "awful, evil hack"), config position wrapped to first
  monitor (`desktop.rb:76–77`). Since Dwin is confined to one desktop, it should
  size/position to *that desktop's current monitor* (which can change as you move
  the desktop between monitors), not to monitor 0 — so this needs to track the
  hosting monitor's geometry, ideally via `ConfigureNotify` (H5).
- **H5.** Ignores `ConfigureNotify` (`mainloop.rb:38`) — when the WM resizes it
  (e.g. desktop 10 moves to a differently-sized monitor), Dwin doesn't re-layout.
  This is the clean way to solve H4: let the WM drive size, react to the notify.
- **H6.** `peek_packet`-based redraw throttling that the code itself says
  "doesn't actually work," with a `@cnt > 10` fallback (`mainloop.rb:24–34`).
  Crude but low-stakes; revisit only if you see repaint storms.
